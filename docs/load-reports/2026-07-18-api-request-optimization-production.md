# Оптимизация API Club PWA: production-проверка 100 клиентов

Дата: 18 июля 2026 года. Среда: production `club2.myn8nservertest.ru`, VPS 2 vCPU / 2 ГБ.

## Итог

Оптимизация внедрена и развёрнута. Production-профиль из 100 независимых HTTP-потоков прошёл полностью: **600 из 600 запросов успешны, ошибок нет, health и readiness сохранили статус 200 до и после нагрузки**.

Фоновая нагрузка одного открытого приложения снижена с пяти до двух запросов в минуту — на **60%**:

| Профиль | На клиента | 150 онлайн | Средний RPS |
|---|---:|---:|---:|
| До оптимизации | 5 запросов/мин | 750 запросов/мин | 12,5 |
| После оптимизации | 2 запроса/мин | 300 запросов/мин | 5,0 |

Здесь учитываются только автоматические фоновые обновления состояния приложения. Действия пользователя, SSE-соединение чата и загрузка медиа считаются отдельно.

## Что изменено

- `/api/me`, `/api/notifications` и `/api/support/unread` заменены одним снимком `/api/app-state`;
- один 30-секундный таймер с небольшим случайным сдвигом заменил три независимых таймера;
- повторный запрос не запускается, пока предыдущий `/api/app-state` не завершён;
- полный список уведомлений загружается только при открытии центра уведомлений;
- список тем сообщества получает данные тремя пакетными запросами вместо схемы `1 + 2N`;
- непрочитанные обращения считаются SQL-агрегацией, без загрузки всех строк;
- `/api/me` больше не повторяет уже выполненный поиск пользователя;
- настройки роли владельца читаются одним запросом.

В синтетическом профиле на 100 клиентов количество запросов уменьшилось с 1200 до 600 — на **50%**, поскольку старый тест одновременно вызывал четыре endpoint, а новый — два.

|---|---:|

По endpoint:

| Endpoint | p50 | p95 | p99 | max |
|---|---:|---:|---:|---:|
| `/api/app-state` | 720 мс | 4943 мс | 5279 мс | 5291 мс |
| `/api/community/topics` | 805 мс | 925 мс | 982 мс | 1006 мс |

Волны:

- первая синхронная волна: 100/100 клиентов, 6077 мс;
- вторая: 100/100, 1382 мс;
- третья: 100/100, 1657 мс.

Высокий p95 сформирован первой искусственно синхронизированной волной из GitHub runner. Это не обычный профиль приложения: в production таймер имеет jitter и клиенты не обращаются к API строго в одну миллисекунду. При этом даже холодная волна не дала ошибок, а следующие две волны завершились примерно за 1,4–1,7 секунды.

## Сравнение с предыдущей проверкой

Предыдущий профиль выполнил 1200 запросов без ошибок: p50 1258 мс, p95 2764 мс, p99 3061 мс. Точное сравнение задержек ограничено различиями генератора и маршрута запуска, но объём запросов сравним напрямую: новый профиль делает вдвое меньше обращений и также не теряет запросы.

## Устойчивость теста

Один промежуточный прогон совпал по времени с production-деплоем и получил `502` в момент штатного перезапуска API-контейнера. Сервер восстановился сразу; чистый повторный прогон прошёл 600/600. Workflow теперь ожидает завершения активного `deploy.yml` до создания временной сессии и начала нагрузки.

Тест использует временную owner-сессию, маскирует токен, отзывает сессию в `always()` и не сохраняет секреты в JSON.

## Вывод для цели 500 в базе / 150 онлайн

Оптимизация сразу уменьшает постоянный фон при 150 открытых приложениях с 12,5 до 5 RPS. Это заметно освобождает текущие 2 vCPU для действий пользователей, чата и загрузок. Production выдержал одновременную волну 100 клиентов без ошибок, но текущий тариф всё ещё нельзя считать гарантией для 150 пользователей, одновременно открывающих тяжёлые экраны: ограничение остаётся по задержке процессора, а не по корректности API.
