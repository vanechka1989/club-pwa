# S3 capacity test: 100 concurrent lesson uploads

## Result

**PASS.** The current production stack completed and verified all 125 test uploads without request failures, health/readiness failures, container restarts, OOM events, or leftover test objects.

- 100 concurrent image clients: 100/100 completed and readable.
- 25 concurrent multipart video clients: 25/25 completed and readable.
- 800 MiB uploaded through the production API/S3 path.
- 675 measured requests, 0 failures, 0% error rate.
- 125/125 completed objects deleted; independent post-run S3 scan found 0 objects containing the run ID.
- API container after the run: healthy, 0 restarts, `OOMKilled=false`.
- Peak sampled API RSS: 463.4 MiB against the 1.5 GB safety limit.

Run ID: `s3-production-100-20260718`

Production host: `https://club2.myn8nservertest.ru`

Execution window: 2026-07-18 04:29:39–04:31:47 UTC.

## Method

The runner used the same production multipart route as the lesson editor:

1. `POST /api/admin/learning/materials/uploads/multipart`;
2. `PUT` each part through the API-proxied upload URL;
3. `POST /api/admin/learning/materials/uploads/multipart/complete`;
4. request an owner-only signed read URL;
5. read byte `0-0` and validate the exact total size from `Content-Range`;
6. delete the object through the owner-only S3 endpoint.

The load generator used one temporary owner session with independent HTTP flows. This intentionally measures concurrent upload/API/S3 capacity, not authentication throughput for 100 different identities. Payloads were deterministic synthetic bytes, and no lesson database records were created.

The production profile had two phases:

| Phase | Clients | Object size | Multipart shape | Total | Ramp |
|---|---:|---:|---:|---:|---|
| Images | 100 | 2 MiB | 1 part | 200 MiB | 5 → 10 → 25 → 50 → 75 → 100 |
| Videos | 25 | 24 MiB | 3 × 8 MiB | 600 MiB | 25 simultaneous |

The image objects were verified and removed before the video phase. Health, readiness, and API metrics were sampled once per second. The run would stop if health/readiness failed or API RSS exceeded 1.5 GB.

The API RSS grew from a 162.5 MiB baseline to a 463.4 MiB sampled peak. This is expected because 8 MiB multipart bodies currently pass through the API process. The peak remained below one third of the configured safety limit and the container stayed healthy.

After hardening the runner so application cookies are sent only to the Club PWA origin, a final production smoke test also passed: 3/3 image objects and 2/2 video objects were uploaded, read through external signed S3 URLs without the application cookie, and removed without residue.

## Capacity conclusion

For the tested workload, the current tariff and production architecture support **100 clients uploading lesson images at the same time**. The heavier video scenario with **25 concurrent 24 MiB uploads** also passed. The observed bottleneck is transfer time through the proxied multipart path, not correctness or API stability.

This result is evidence for the exact tested sizes and concurrency, not an unlimited-capacity guarantee. Larger files, 100 simultaneous multi-part videos, a slower client uplink, or concurrent chat/API traffic require a separate mixed-load run. The repeatable runner and thresholds are now part of the repository so the same test can be rerun after VPS, S3, proxy, or upload-pipeline changes.
