# Нагрузочная проверка Club PWA: 100 одновременных клиентов

Дата прогона: 18 июля 2026 года. Среда: рабочий production `club2.myn8nservertest.ru`, текущий VPS 2 vCPU / 2 ГБ. Проверялись SSE-чаты и связанные авторизованные API-запросы. Тестовые тема, сообщения и сессия после каждого прогона удалены.

## Итог

**Realtime на 100 одновременных соединений пройден. Общий нагрузочный профиль пройден условно.**

- 100 из 100 SSE-клиентов подключились;
- 100 соединений непрерывно удерживались 180 секунд, обрывов не было;
- доставлено 2000 из 2000 событий залпа, потерь и дублей нет;
- 25 из 25 принудительно отключённых клиентов переподключились;
- 1200 параллельных API-запросов завершились без ошибок;
- API, PostgreSQL, web и Caddy не перезапускались, OOM не было;
- не пройден только целевой порог времени ответа API при синхронном всплеске 100 клиентов: p95 2764 мс вместо 500 мс.

Это означает, что **100 пользователей могут одновременно находиться онлайн и получать сообщения в реальном времени**. Если все 100 одновременно начнут обновлять тяжёлые экраны, приложение продолжит работать, но ответы будут заметно медленнее.

## Сценарий

1. Создана временная скрытая тема и временная авторизованная сессия владельца.
2. Внешние клиенты подключались к production через HTTPS/SSE ступенями 10 → 25 → 50 → 75 → 100.
3. На 100 клиентах соединения удерживались 180 секунд с регулярной проверкой `/api/health`, `/api/ready`, `/api/metrics`.
4. Отправлено 10 последовательных сообщений: 1000 измерений доставки.
5. Одновременно отправлено 20 сообщений: 2000 ожидаемых доставок.
6. Принудительно переподключено 25 клиентов и проверено следующее сообщение.
7. Выполнено 1200 запросов: по 100 клиентов, четыре рабочих endpoint, три раунда.
8. Соединения закрыты, сообщения удалены, временная тема и сессия удалены.

Каждое SSE-соединение было самостоятельным сетевым клиентом, но использовало одну временную сессию владельца. Это корректно измеряет стоимость соединений, авторизации, рассылки событий и API-нагрузки, но не моделирует разнообразие ролей ста разных аккаунтов.

Постоянные SSE-подключения не удерживают отдельные соединения PostgreSQL. Память увеличивается умеренно; аварийного роста, OOM и рестартов нет. Значения container memory сняты точечными серверными замерами, поэтому это не непрерывный абсолютный максимум.

## Что проверено дополнительно

- production-защита требует явное `CONFIRM_PRODUCTION_LOAD=YES`;
- есть аварийная остановка при unhealthy/readiness, RSS API выше 1,5 ГБ и доле HTTP-ошибок выше 1%;
- отчёт не содержит cookie или токен сессии;
- cleanup выполняется в `finally` даже при ошибке;
- после теста `realtimeSubscribers = 0`;
- в БД не осталось тем с именем `LOAD TEST%`;
- все четыре production-контейнера остались в состоянии `running`, restart count = 0.

## Автоматическая проверка проекта

Перед фиксацией результатов дополнительно выполнены проверки всего монорепозитория:

- `pnpm check` — TypeScript-проверки shared, API и web прошли;
- `pnpm test` — весь набор тестов прошёл, включая 86 API-файлов / 250 API-тестов и 82 web-файла / 599 web-тестов;
- `pnpm build` — production-сборка web прошла (1776 модулей); остался только существующий неблокирующий warning Vite о чанке больше 500 КБ;
- специализированный набор нагрузки — 2 файла / 9 тестов прошли;
- синтаксис `community-100.mjs` и `git diff --check` прошли.

Локально `k6` не установлен, поэтому production-прогон выполнен самостоятельным Node-сценарием. Для SSE он информативнее текущего k6-профиля: каждое соединение имеет отдельный поток, парсер событий и точный учёт доставки, дублей, переподключений и задержек fan-out.

## Ограничения проверки

- использована одна временная identity, поэтому отдельно не проверялись сто разных ролей, тарифов и персональных наборов прав;
- не проверялись загрузка крупных файлов, email, push и внешняя форма Prodamus: они не относятся к доставке сообщений SSE;
- тест не доказывает готовность к 150 одновременно активным пользователям;
- маршрутизация между изолированными личными комнатами не проверялась, поскольку текущая модель сообщества использует общий поток `community.changed` с `topicId`, а не отдельный websocket-room на пользователя.

## Вывод по вместимости

Для цели **500 зарегистрированных клиентов и до 100 одновременно онлайн** текущая конфигурация подходит, если поведение обычное: чтение, переписка, отдельные переходы по экранам. Realtime имеет хороший запас по памяти и не теряет события.

Для **150 одновременно активных клиентов** на текущем тарифе гарантию давать нельзя. Перед таким обещанием нужно:

1. снизить p95 API-всплеска — кэшировать частые счётчики, объединить стартовые запросы или ограничить одновременное обновление четырёх endpoint;
2. провести ступенчатый тест 100 → 125 → 150 после оптимизации;
3. при сохранении p95 выше 1–1,5 секунды перейти минимум на 4 vCPU / 4–8 ГБ либо масштабировать API в несколько экземпляров с Redis/pub-sub для realtime.

Текущий production не упал и данные не потерял, но тест показывает: первым ограничением станет не память и не число SSE-соединений, а задержка API при синхронных запросах.
