# Аналитика вовлечённости рассылок — спецификация

## Цель

Добавить в историю рассылок достоверную аналитику доставки и вовлечённости по Push и Email: уникальные открытия, уникальные переходы, проценты, динамику, ссылки и получателей по статусам. Отслеживание применяется только к новым рассылкам после выпуска версии 5.26; старые рассылки продолжают показывать существующие показатели доставки.

## Выбранный подход

Используется серверный сбор событий с подписанными HMAC-токенами. Для Email HTML получает пиксель открытия и подписанные ссылки-переходы. Для системного Push нажатие ведёт через подписанный серверный URL в центр уведомлений, а чтение рассылки внутри приложения также фиксирует открытие. События открытия и перехода по конкретной ссылке идемпотентны для каждого получателя, поэтому повторные загрузки пикселя и повторные клики не завышают число уникальных пользователей.

Альтернативы отклонены:

- только показатели доставки не показывают вовлечённость;
- аналитика без Email-пикселя не даёт показатель открытий, который пользователь явно выбрал;
- сторонняя аналитическая платформа добавила бы внешнюю передачу данных и инфраструктурную зависимость без необходимости.

## Модель данных

В `admin_mailings` добавляется `analytics_enabled_at`. Для существующих записей значение остаётся `NULL`; для новых рассылок оно устанавливается при создании.

Создаётся таблица `admin_mailing_events`:

- `id` — UUID;
- `mailing_id` — рассылка с каскадным удалением;
- `recipient_id` — строка доставки с каскадным удалением;
- `event_type` — `open` или `click`;
- `event_key` — `open` либо `click:<sha256 URL>`;
- `destination` — исходный HTTP/HTTPS URL только для клика;
- `occurred_at` — время первого события.

Уникальный индекс `(recipient_id, event_key)` гарантирует одну запись открытия и одну запись перехода по каждой уникальной ссылке на получателя. Индексы `(mailing_id, event_type, occurred_at)` и `(mailing_id, occurred_at)` обеспечивают агрегацию. IP, User-Agent, cookies и технические отпечатки не сохраняются.

## Подписанные ссылки и безопасность

Токен содержит версию, назначение (`open`, `click` или `push`), UUID получателя и, для клика, URL назначения. Полезная нагрузка кодируется Base64URL и подписывается HMAC-SHA256 с доменным префиксом `mailing-tracking:v1`, используя уже настроенный `MAILING_UNSUBSCRIBE_SECRET`. Подпись сравнивается через `timingSafeEqual`.

Сервер принимает только корректно подписанные токены. Redirect разрешён только на `http:` и `https:`. `mailto:`, URL отписки и неподдерживаемые схемы не переписываются. Неподписанный URL не может превратить endpoint в открытый редирект.

## Сбор событий

### Email

Перед отправкой уже очищенный HTML обрабатывается сервером:

- HTTP/HTTPS-ссылки заменяются на `/api/mailings/track/click?token=...`;
- ссылка вложения также отслеживается;
- в конец HTML добавляется невидимый пиксель `/api/mailings/track/open?token=...`;
- текстовая версия письма остаётся читаемой и не получает скрытый пиксель.

Endpoint открытия записывает событие и возвращает прозрачный GIF с `Cache-Control: no-store, private`. Email-открытия в интерфейсе явно помечаются как приблизительные: блокировка картинок, почтовые прокси и автоматическая загрузка изображений влияют на показатель.

Endpoint клика сначала проверяет токен, затем идемпотентно записывает событие и отвечает `302` на исходный адрес. Для корректного токена временная ошибка записи логируется, но не лишает пользователя перехода.

### Push и уведомления приложения

Web Push новой рассылки получает URL вида `/api/mailings/track/push?token=...`. Нажатие фиксирует открытие и переводит пользователя на `/notifications`. Когда пользователь открывает центр уведомлений без нажатия Push, существующие endpoints чтения фиксируют открытия для непрочитанных уведомлений с `source = mailing` до изменения `read_at`. Повторная фиксация безопасна благодаря уникальному индексу.

## API администратора

`GET /admin/mailings/:id/analytics` возвращает:

- `trackingEnabledAt` и пояснение о приблизительности Email-открытий;
- сводку: доставлено, уникально открыто, уникально перешли, open rate, CTR и CTOR;
- разбивку Push/Email с доставкой, ошибками, пропусками, открытиями и кликами;
- временные интервалы с доставкой, открытиями и кликами;
- ссылки с URL и числом уникальных переходов.

Проценты рассчитываются так:

- `openRate = uniqueOpened / sentCount`;
- `clickRate = uniqueClicked / sentCount`;
- `clickToOpenRate = uniqueClicked / uniqueOpened`.

При нулевом знаменателе возвращается `0`. Один получатель с несколькими ссылками считается один раз в общем `uniqueClicked`, но по одному разу в статистике каждой ссылки.

`GET /admin/mailings/:id/recipients?status=...&channel=...&limit=50&cursor=...` возвращает получателей с каналом, состоянием доставки, числом попыток, безопасным текстом ошибки и временами отправки, открытия и первого перехода. Статусы фильтра: `all`, `delivered`, `opened`, `clicked`, `failed`, `skipped`, `pending`. Действующие права раздела рассылок применяются без изменений.

## Интерфейс

В карточке подробностей рассылки появляется раздел «Аналитика»:

- KPI-карточки «Доставлено», «Открыли», «Перешли», «Open rate», «CTR», «CTOR»;
- компактная разбивка Push/Email;
- график динамики доставки, открытий и переходов;
- список популярных ссылок;
- фильтры получателей по статусу и каналу с постраничной загрузкой.

Для старой рассылки показывается сообщение «Отслеживание вовлечённости появилось в версии 5.26» без ложных нулевых процентов. Для Email рядом с открытиями выводится короткая пометка «приблизительно».

Интерфейс остаётся mobile-first: KPI переносятся, таблицы превращаются в карточки, фильтры и кнопки имеют область нажатия не меньше 44 px, горизонтальная прокрутка страницы не появляется.

## Ошибки и эксплуатация

- Ошибка трекинга не меняет состояние доставки и не запускает повторную отправку.
- Неверный токен возвращает `404` без раскрытия причины.
- Повторное событие отвечает успешно, но не создаёт новую строку.
- Аналитика строится из фактических строк доставки и событий, а не из накапливаемых счётчиков.
- В список получателей не возвращаются секреты, токены или полный стек ошибок.
- Миграция обязательно регистрируется в Drizzle journal и проверяется тестом.

## Тестирование и выпуск

Тесты покрывают HMAC-токены, запрет открытого редиректа, переписывание HTML, идемпотентность событий, фиксацию чтения Push, формулы агрегации, фильтры получателей, схемы общего пакета и адаптивную разметку. После целевых тестов выполняются полные `pnpm test`, `pnpm build` и `git diff --check`.

Выпуск: приложение 5.26, Service Worker cache `club-pwa-v216`. После развёртывания проверяются commit сервера, миграция и индексы, health API, публичные tracking endpoints с неверным токеном, версия PWA, журналы API и адаптивность production на 320×720, 390×844, 768×1024, 1024×768 и 1440×900.
