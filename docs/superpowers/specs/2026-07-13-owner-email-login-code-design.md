# Аварийный код входа по email — дизайн

## Задача

Дать владельцу клуба возможность помочь существующему клиенту войти, если письмо с кодом не доставляется. Владелец вводит email, получает одноразовый шестизначный код и передаёт его клиенту вне приложения. Клиент использует обычный экран входа по email.

В текущей модели роль `owner` отображается как «Главный админ». Только эта роль получает функцию. Обычный `admin`, включая администратора с разрешением `project_settings`, не получает доступ ни в интерфейсе, ни через прямой API-запрос.

## Рассмотренные варианты

1. **Создавать обычный код в существующей таблице авторизации — выбранный вариант.** Код проходит штатную проверку, одноразово поглощается и не создаёт отдельного обходного способа входа.
2. Создавать готовую сессию или ссылку входа. Это сложнее безопасно передавать, а утечка ссылки сразу даёт доступ без знания email и кода.
3. Разрешить владельцу задавать произвольный постоянный код. Это создаёт слабый повторно используемый секрет и не подходит.

## Серверная логика

- Новый owner-only endpoint принимает email и нормализует его тем же способом, что обычная авторизация.
- Генерация разрешена только для уже существующего пользователя с этим email. Неизвестный email возвращает понятную ошибку и не создаёт аккаунт.
- Сервер создаёт шестизначный код через существующий криптографический генератор `createLoginCode()`.
- Перед вставкой все прежние неприменённые коды этого email помечаются использованными.
- Новый код хранится только как SHA-256-хеш связки `email:code` в существующей таблице `auth_email_login_codes`.
- Срок действия совпадает с текущим `AUTH_LOGIN_CODE_TTL_MINUTES` (сейчас 10 минут).
- Ответ содержит нормализованный email, код и точное время истечения. Код не пишется в логи и показывается интерфейсом только в текущем результате запроса.
- Действие записывается в `admin_action_logs`: исполнитель, клиент, email, время истечения. Сам код в metadata отсутствует.
- Повторная генерация защищена серверным интервалом 30 секунд для одного email; интерфейс также блокирует повторное нажатие во время запроса.

## Интерфейс

В разделе «Настройки проекта» под реферальной системой появляется компактная карточка «Аварийный вход по email», только при `session.user.realRole === "owner"`.

Состояния карточки:

1. Поле email и кнопка «Сгенерировать код».
2. Во время запроса кнопка заблокирована и показывает «Генерируем…».
3. После успеха показываются email, крупный шестизначный код, срок действия и кнопка «Скопировать код».
4. Кнопка «Создать другой» очищает показанный результат и возвращает форму.
5. Ошибки «клиент не найден», неверный email, слишком частая генерация и ошибка сервера показываются внутри карточки.

Копирование не включает лишний текст — только шесть цифр. После обновления страницы код больше не отображается, но остаётся действительным до входа или истечения.

## Безопасность

- Проверка owner выполняется сервером до чтения входных данных и генерации.
- UI-ограничение является только дополнительным удобством и не заменяет серверную авторизацию.
- Код одноразовый, ограничен временем, привязан к конкретному email и аннулирует предыдущие коды.
- API не сообщает данные других клиентов, кроме подтверждения найденного email.
- В журнале действий нет кода или его хеша.

## Проверка

- Unit/source-тесты подтверждают owner-only guard, отсутствие кода в аудите, аннулирование прежних кодов и совместимость со штатной проверкой.
- Web-тесты подтверждают скрытие карточки от `admin`, отображение владельцу, состояние загрузки и результат.
- Полные `pnpm test`, `pnpm check`, `pnpm build`.
- Мобильная визуальная проверка карточки на ширине 360 px.
- Production-проверка версии, service worker, health, commit и отказ API для не-owner сессии, если безопасная тестовая сессия доступна.
