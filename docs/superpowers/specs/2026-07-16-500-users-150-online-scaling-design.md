# Масштабирование Club PWA до 500 клиентов и 150 одновременных пользователей

**Дата:** 2026-07-16
**Цель:** обеспечить 500 зарегистрированных клиентов и 150 одновременно активных пользователей без потери realtime-сообщений, дублирования фоновых задач и лавинообразной нагрузки на PostgreSQL.

## Текущее состояние и узкие места

- production-сервер имеет 2 vCPU и 2 ГБ RAM; API, PostgreSQL, Caddy и web работают на одном узле;
- запущен один экземпляр API с пулом PostgreSQL на 10 соединений;
- каждый авторизованный запрос обновляет `auth_sessions.last_seen_at`, создавая лишние записи и блокировки;
- клиент одновременно опрашивает платежи, сессию, поддержку и уведомления каждые 10–15 секунд;
- community realtime хранит подписчиков только в памяти процесса, поэтому второй API не увидит события первого;
- фоновые очистки и диспетчер рассылок запускаются внутри каждого API-процесса и при масштабировании будут дублироваться;
- API обращается напрямую к PostgreSQL, без ограничения общего числа соединений нескольких реплик;
- нет воспроизводимого нагрузочного сценария для 150 активных клиентов.

## Выбранная архитектура

### Ресурсы production

Целевой сервер: **8 vCPU, 16 ГБ RAM, SSD от 80 ГБ**. Программная часть не будет считать текущие 2 vCPU/2 ГБ достаточными для гарантированных 150 активных пользователей. Изменение тарифа/VPS выполняется у провайдера отдельно от репозитория.

### Контейнеры

- `web` и `caddy` остаются stateless;
- `api-1` и `api-2` обслуживают HTTP и SSE;
- `worker` один раз запускает диспетчер рассылок и фоновые очистки, но не принимает внешний трафик;
- `redis` обеспечивает межпроцессный pub/sub realtime-событий;
- `pgbouncer` принимает соединения API и ограничивает их число перед PostgreSQL;
- `postgres` хранит основные данные;
- `migrate` подключается напрямую к PostgreSQL, а не через transaction pooling PgBouncer.

### Realtime

Публикация community-события выполняется одновременно локально и через Redis. Каждая API-реплика подписана на общий Redis-канал и доставляет событие своим SSE-клиентам. Идентификатор события — UUID, поэтому события не конфликтуют между процессами. При временной недоступности Redis локальная доставка продолжает работать, а reconnect выполняется автоматически.

### Авторизация и активность

- проверка действующей сессии остаётся в PostgreSQL, чтобы отзыв доступа применялся сразу;
- `last_seen_at` обновляется не чаще одного раза в пять минут на сессию;
- история IP записывается только при реальной смене IP, как и сейчас;
- проверка owner/preview выполняется только если клиент действительно прислал preview-заголовок;
- это сокращает число записей в БД и лишних owner-запросов без ослабления контроля доступа.

### Клиентский фон

- все фоновые циклы работают только при видимой странице;
- проверки сессии, поддержки и уведомлений выполняются реже и со случайным jitter, чтобы 150 клиентов не отправляли запросы в одну миллисекунду;
- при возврате в приложение данные обновляются сразу;
- открытая community-тема использует SSE, polling остаётся только резервным режимом после разрыва realtime.

### Надёжность

- `/health` остаётся быстрым liveness endpoint;
- `/ready` проверяет PostgreSQL и Redis и используется для готовности реплики;
- worker отделён от API, поэтому горизонтальное масштабирование не запускает одну рассылку несколько раз;
- Caddy балансирует запросы между двумя API-репликами;
- Redis и PgBouncer имеют healthcheck и постоянный restart policy.

## Нагрузочный профиль и критерии приёмки

Сценарий содержит:

- постепенный рост до 150 виртуальных пользователей;
- 100 активных пользователей, читающих основные экраны и уведомления;
- 30 участников community с открытым SSE;
- 20 администраторских/интенсивных сессий;
- удержание пика не менее 15 минут.

Критерии:

- HTTP error rate меньше 0,5%;
- p95 основных API-запросов меньше 500 мс, p99 меньше 1,5 с;
- realtime-событие доставляется 95% клиентов меньше чем за 1 секунду;
- PostgreSQL не превышает 70 активных backend-соединений;
- нет дублирования рассылок и фоновых заданий;
- память каждого API остаётся стабильной, без устойчивого роста после завершения теста.

## Развёртывание

Пользователь решил пока сохранить текущий тариф 2 vCPU/2 ГБ. Поэтому текущий production остаётся на одной API-реплике с прямым подключением к PostgreSQL. В этом релизе включаются только безопасные оптимизации auth/polling, `/ready`, метрики и разделение фоновых заданий без дополнительных контейнеров.

Полная схема с Redis, PgBouncer, worker и второй API-репликой сохранена в `docker-compose.scale.yml`, но не запускается и не расходует ресурсы текущего VPS. Её активация допускается только после увеличения VPS и проверки `docker compose config`.

Нагрузочный сценарий на 150 пользователей также подготовлен, но на текущем production не запускается. После изменения тарифа тест выполняется ступенями 50, 100 и 150 пользователей.

## Не входит в релиз

- перенос PostgreSQL на отдельный managed-сервис;
- Kubernetes;
- гарантии для 150 активных пользователей на текущем VPS 2 vCPU/2 ГБ;
- online/presence и индикатор «печатает» — они не нужны для выбранного объёма и создают дополнительный трафик.
