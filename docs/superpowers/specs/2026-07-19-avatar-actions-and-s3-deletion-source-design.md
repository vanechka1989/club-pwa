# Действия с аватаром и источник удалённого S3-файла

## Цель

Сделать управление фотографией профиля заметным и предсказуемым, а записи об удалении файлов из S3 — понятными без расшифровки технического пути.

## Профиль

- Кнопка камеры переносится с аватара в строку имени и располагается справа от карандаша изменения ника.
- Аватар становится кнопкой с полноценной областью нажатия и доступным названием.
- При существующей фотографии нажатие на аватар или камеру открывает единое меню с действиями «Загрузить новое фото» и «Настроить кадр».
- При отсутствии фотографии оба действия открывают системный выбор изображения.
- Существующие загрузка, проверка файла, кадрирование и сохранение позиции не изменяются.

## Источник удалённого файла

Серверный resolver получает S3-ключ и возвращает структурированное описание источника:

- категория и человекочитаемый тип файла из S3-пути;
- название урока и, для вложенного материала, название карточки;
- название темы и вид вложения для сообщений общения;
- тему обращения поддержки;
- название рассылки или уведомления;
- признак того, что точную связь найти не удалось.

Перед новым ручным удалением объект сначала разрешается в источник. Снимок источника записывается в `metadata` административного события вместе с S3-ключом, поэтому подпись сохраняется даже после удаления связанных данных.

Для старых событий `storage.s3.object.deleted`, у которых снимка ещё нет, endpoint истории группирует ключи и пытается восстановить источники по текущим связям базы. Если точная связь уже потеряна, остаётся классификация по S3-пути.

## Отображение истории

Основной заголовок записи остаётся кратким: «Удалён файл из S3». Ниже выводятся:

- `Источник: урок «…» · карточка «…»`;
- `Источник: общение · тема «…» · голосовое сообщение`;
- `Источник: поддержка · обращение «…»`;
- `Источник: рассылка «…»` или `уведомление «…»`;
- при неизвестной связи — общая категория и тип файла.

Технический S3-путь показывается отдельной приглушённой строкой, чтобы он оставался доступен для диагностики и не конкурировал с понятным описанием.

## Архитектура и поток данных

1. Общий серверный resolver запрашивает связанные записи пакетно и строит карту по S3-ключу.
2. Endpoint удаления запрашивает источник до вызова удаления и сохраняет снимок в аудит.
3. Endpoint истории использует сохранённый снимок, а для старых записей пакетно вызывает resolver.
4. API продолжает возвращать совместимую схему `AdminActionLog`; структурированные поля находятся в существующем `metadata`.
5. Frontend-презентер безопасно читает метаданные и формирует отдельные строки источника и пути.

## Ошибки и обратная совместимость

- Ошибка определения источника не блокирует удаление: сохраняются классификация пути и ключ.
- Неизвестные или повреждённые метаданные игнорируются без ошибки экрана.
- Старые журнальные записи без ключа отображаются как раньше.
- Секреты S3 и подписанные URL в журнал не попадают.

## Проверка

- Компонентные тесты подтверждают расположение камеры и одинаковое действие аватара и камеры.
- Серверные тесты покрывают урок, вложенную карточку, чат, поддержку, рассылку, уведомление и неизвестный ключ.
- Тест endpoint удаления проверяет сохранение снимка до удаления.
- Тест истории проверяет дополнение старой записи.
- Frontend-тесты проверяют понятную подпись источника, технический путь и fallback.
- Визуальная проверка профиля выполняется на мобильной и широкой ширине без перекрытия имени и горизонтального скролла.
