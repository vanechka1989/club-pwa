# Компактный профиль и единый отображаемый ник

## Цель

Пересобрать пользовательский профиль в компактной mobile-first компоновке и добавить единый уникальный отображаемый ник. Ник должен одинаково отображаться в профиле, сообществе, ответах, закреплениях, поддержке и административных экранах без изменения существующей авторизации.

## Компоновка профиля

- Шапка сохраняет общий `PageHeader` приложения.
- Основная карточка использует крупный аватар и горизонтальный блок пользователя из утверждённого варианта 2.
- Рядом с ником располагается действие редактирования, пока самостоятельная смена доступна.
- Email выводится отдельной компактной строкой.
- Статус доступа объединяется в одну секцию:
  - слева — название активного тарифа либо «Ручной доступ»;
  - справа — дата окончания;
  - ниже — тонкий прогресс и количество оставшихся дней.
- Отдельные дублирующие плашки «Подписка активна», дата и блок «Ручной доступ» удаляются.
- «Продлить» остаётся основным действием, «Выйти» — вторичным.
- Прогресс обучения отображается одной компактной строкой.
- Все элементы используют существующие токены, четыре темы, safe-area и минимальную область нажатия 44 px.

## Модель отображаемого имени

- В таблицу пользователей добавляется отдельное nullable-поле `display_name`.
- Технические `username`, email, Telegram ID и логика авторизации не изменяются.
- Общий приоритет отображения имени: `displayName`, затем `firstName`, затем `username`, затем ID.
- Общая функция форматирования применяется во всех пользовательских и административных разделах.
- API-схемы пользователя и автора сообщения получают `displayName`.
- Исторические сообщения не копируют новое имя в текст: при загрузке они получают актуальный `displayName` автора, поэтому после смены ника старые сообщения также показывают новое имя.

## Правила ника

- Длина: от 3 до 20 символов после trim.
- Разрешены русские и латинские буквы, цифры, `_` и `-`.
- Уникальность проверяется без учёта регистра.
- Сервер является единственным источником проверки и сохранения.
- Занятый или некорректный ник не сохраняется; форма показывает локализованную ошибку под полем.
- Для защиты от гонки в базе создаётся нормализованное уникальное значение или функциональный уникальный индекс по нижнему регистру.

## Одноразовая самостоятельная смена

- Пользователь может самостоятельно сохранить новый ник ровно один раз за всё время.
- Факт использования хранится на сервере отдельным timestamp `display_name_changed_by_user_at`.
- Изначально существующий технический username не считается самостоятельной сменой.
- После успешной смены сервер навсегда запрещает повторное пользовательское изменение.
- После использования права карандаш скрывается; интерфейс показывает «Изменение доступно через администратора».
- Административная смена не очищает timestamp и не возвращает право пользователю.

## Пользовательский сценарий

1. Пользователь нажимает карандаш рядом с ником.
2. Открывается нижняя панель «Изменить ник» с текущим значением, правилами и кнопками «Отмена»/«Сохранить».
3. Клиент выполняет базовую проверку формата, сервер проверяет формат, уникальность и право смены.
4. После успеха session user обновляется без перезагрузки, панель закрывается, профиль показывает новый ник.
5. Следующая загрузка чата и других экранов получает тот же `displayName` из API.

## Административный сценарий

- В карточке клиента добавляется секция «Ник» с текущим значением и действием «Изменить».
- Доступ разрешён администраторам, у которых уже есть право управления клиентами.
- Администратор не ограничен одноразовым правилом, но соблюдает формат и уникальность.
- Каждая административная смена записывается в существующий журнал действий: клиент, старое значение, новое значение, администратор и время.
- Ошибка уникальности отображается внутри формы, без системных alert.

## API и поток данных

- Пользовательский endpoint изменяет только ник текущего пользователя и проверяет одноразовое право.
- Административный endpoint изменяет ник выбранного клиента и требует право управления клиентами.
- Ответы endpoint возвращают обновлённого пользователя/клиента.
- Клиент обновляет session store после пользовательской смены и выбранную карточку после административной смены.
- Автор сообщения, ответ, закрепление и тикет используют общий mapper отображаемого имени.

## Ошибки

- `400` — некорректный формат.
- `403` — самостоятельная смена уже использована или нет административного права.
- `409` — ник занят.
- `404` — клиент не найден в административном сценарии.
- Интерфейс показывает понятные inline-сообщения и сохраняет введённое значение.

## Проверки

- Миграция базы и уникальность без учёта регистра.
- Валидация допустимых и недопустимых символов.
- Первая пользовательская смена проходит, вторая отклоняется.
- Административная смена проходит после пользовательской и не восстанавливает право.
- Новый ник появляется в session/profile, новых и исторических сообщениях, ответах, закреплениях, поддержке и карточке клиента.
- Компактный профиль проверяется на 320, 360, 390, 412 и 768 px во всех четырёх темах.
- Проверяются длинный email, максимальный ник, активный/истёкший доступ, отсутствие тарифа, loading/error/disabled состояния.

## Вне объёма

- Изменение email, Telegram username, ID, авторизации или ролей.
- История публичных прежних ников.
- Возврат права самостоятельной смены администратором.
