# PWA UI Foundation Design

**Дата:** 2026-07-11  
**Статус:** утверждён пользователем для реализации  
**Источник требований:** запрос на перевод всего PWA на единый foundation по `pwa-ui-master`

## Цель

Перевести приложение на единую mobile-first UI-основу без изменения бизнес-логики, API, маршрутов, ролей, платежей, отправки сообщений, загрузок файлов, PWA/service worker/manifest поведения и структуры данных.

Результат должен выглядеть как одно аккуратное приложение: одинаковые контейнеры, шапки, карточки, кнопки, поля, нижняя навигация, нижние панели действий, состояния загрузки/пусто/ошибка и корректная работа на мобильных экранах от 320 px.

## Диагноз текущей проблемы

Проблема не сводится к кешу или одному экрану. В текущем UI накопилось несколько поколений глобального CSS и layout-решений:

- один большой `apps/web/src/styles.css` содержит тысячи строк и много повторных определений одних и тех же сущностей;
- классы вроде `.section-head`, `.soft-card`, `.surface-card`, `.icon-button`, `.bottom-nav`, `.task-screen`, `.primary-button`, `.secondary-button` переопределяются в разных местах;
- часть экранов использует собственные контейнеры, шапки, карточки и кнопки;
- локальные правила перебивают системные размеры, из-за чего появляются мизерные action-кнопки, разные отступы и разные высоты;
- в CSS есть несколько конфликтующих theme/layout слоёв;
- fixed/sticky/overflow/min-height правила местами создают пустые области, наслаивание и проблемы с мобильной клавиатурой;
- существующие E2E-проверки ловят часть геометрических ошибок, но не закрепляют единый визуальный foundation.

## Выбранный подход

Выбран подход **foundation-first migration**.

Не переписываем бизнес-логику приложения. Пересобираем UI-слой вокруг единого foundation и постепенно переводим все экраны на него. Старые конфликтующие CSS-правила удаляются или изолируются только после того, как соответствующий экран перешёл на новые базовые компоненты.

Этот подход выбран вместо:

1. **Быстрого CSS-патча.** Он быстрее, но продолжит множить конфликтующие override-слои.
2. **Полного удаления приложения и переписывания с нуля.** Это может быстрее дать красивый макет, но с высоким риском сломать оплату, поддержку, чаты, роли, API и PWA.

## Архитектура foundation

Создаётся единый UI foundation внутри веб-приложения:

```text
apps/web/src/features/ui/
  foundation.css
  index.ts
  UiAppShell.vue
  UiPageContainer.vue
  UiPageHeader.vue
  UiPageSection.vue
  UiCard.vue
  UiButton.vue
  UiIconButton.vue
  UiButtonGroup.vue
  UiFormField.vue
  UiResponsiveGrid.vue
  UiBottomActionBar.vue
  UiEmptyState.vue
  UiLoadingState.vue
  UiErrorState.vue
```

`foundation.css` становится источником дизайн-токенов и базовых классов. Vue-компоненты задают одинаковую структуру и размеры. Экранные компоненты отвечают только за данные, сценарии, тексты и уникальные композиции.

## Дизайн-токены

Foundation должен использовать семантические токены, а не локальные цвета и размеры:

- `--page-max-width: 768px`
- `--page-padding: 16px`
- `--page-padding-compact: 12px`
- `--section-gap: 24px`
- `--card-gap: 16px`
- `--card-padding: 20px`
- `--card-padding-compact: 16px`
- `--card-radius: 20px`
- `--card-radius-compact: 18px`
- `--control-height: 48px`
- `--button-height: 48px`
- `--button-height-large: 52px`
- `--button-height-compact: 44px`
- `--icon-button-size: 44px`
- `--icon-button-size-compact: 44px`
- `--header-min-height: 76px`
- `--bottom-nav-height: 76px`
- `--bottom-action-height: 72px`
- `--safe-bottom: env(safe-area-inset-bottom, 0px)`
- `--safe-top: env(safe-area-inset-top, 0px)`
- `--motion-fast: 140ms`
- `--motion-normal: 220ms`

Размеры строятся на 8 px сетке с допустимыми 4 px и 12 px оптическими шагами.

## Темы

Foundation обязан поддерживать четыре варианта:

1. Dark Soft Touch — День.
2. Dark Soft Touch — Ночь.
3. Graphite + Electric Blue — День.
4. Graphite + Electric Blue — Ночь.

Темы задаются через семантические токены:

- фон приложения;
- фон страницы;
- поверхность карточки;
- поверхность панели;
- границы;
- основной текст;
- вторичный текст;
- muted text;
- primary;
- primary hover/active;
- danger;
- success;
- warning;
- focus ring;
- disabled surface/text;
- shadow.

Локальные жёсткие цвета в экранных компонентах запрещаются, кроме случаев уникального контента, где цвет всё равно должен идти через токен.

## Layout

Все маршруты должны использовать общий shell:

```text
UiAppShell
  UiPageContainer
    UiPageHeader
    UiPageSection / контент маршрута
    UiBottomActionBar при необходимости
  BottomNavigation при необходимости
```

Контейнер страницы:

```css
width: 100%;
max-width: 768px;
margin-inline: auto;
min-width: 0;
```

Контент:

```css
padding-inline: var(--page-padding);
padding-top: 16px;
padding-bottom: calc(24px + var(--safe-bottom));
```

На ширине до 360 px:

```css
padding-inline: var(--page-padding-compact);
```

`min-height: 100dvh` допустим только для общего shell. Карточки, формы, списки, уроки, тикеты и разделы не должны искусственно растягиваться на высоту viewport.

## Шапка страницы

`UiPageHeader` заменяет разрозненные шапки.

Состав:

- back action 44 x 44 px, если нужен;
- текстовый блок `min-width: 0; flex: 1`;
- заголовок с переносом;
- подзаголовок с переносом;
- action slot справа.

Правила:

- кнопка назад не уходит в отдельную строку;
- длинные email, id, названия и подзаголовки переносятся;
- action справа не сжимает заголовок до нечитаемого состояния;
- на узких экранах action может перейти на вторую строку;
- высота шапки не фиксируется жёстко.

## Карточки

`UiCard` становится единой карточкой:

- `width: 100%`;
- `min-width: 0`;
- `height: auto`;
- `padding: var(--card-padding)`;
- `border-radius: var(--card-radius)`;
- фон, граница и тень только через токены.

На ширине до 360 px карточка использует compact padding/radius.

Связанные данные объединяются в секции, а не разбиваются на много больших независимых карточек. Вложенность карточек допускается только когда это реально разные смысловые поверхности, но не как способ добавить отступы.

## Кнопки и icon-кнопки

`UiButton`:

- обычная высота 48 px;
- крупная высота 52 px;
- compact высота 44 px;
- `min-width: 0`;
- подпись не выходит за экран;
- loading и disabled состояния читаемые во всех темах.

`UiIconButton`:

- минимум 44 x 44 px;
- одинаковый радиус;
- одинаковый icon-size;
- accessible name обязателен;
- варианты: neutral, primary, ghost, danger.

Мизерные кнопки добавления, редактирования, назад, меню, камеры, crop, стрелок и удаления заменяются на `UiIconButton` или получают системный класс foundation.

## Формы

`UiFormField` задаёт одинаковый label/help/error layout.

Поля:

- `width: 100%`;
- `min-width: 0`;
- `min-height: 48px`;
- `font-size: 16px` на мобильных;
- textarea `min-height: 120px; resize: vertical`;
- disabled состояние читаемое;
- checkbox/radio строки кликабельные.

Формы на мобильных идут одной колонкой. Две колонки допускаются только на ширине, где содержимое реально помещается.

## Нижняя навигация и нижние панели действий

Bottom navigation:

- находится внутри ширины page container;
- учитывает `safe-area-inset-bottom`;
- не перекрывает последний элемент контента;
- скрывается или переводится в безопасный режим там, где экран использует полноэкранную форму или нижнюю action-панель.

`UiBottomActionBar`:

- предпочтительно `position: sticky; bottom: 0`;
- если используется fixed, контент получает padding под полную высоту панели;
- кнопки переходят в одну колонку на узком экране;
- при открытой мобильной клавиатуре не перекрывает активное поле.

## Мобильная клавиатура

Формы, тикеты, чат, создание рассылки, поиск и textarea должны работать с visual viewport:

- активное поле остаётся видимым;
- страница скроллится основным scroll, а не случайной внутренней карточкой;
- fixed-панели не остаются поверх клавиатуры без компенсации;
- `100vh` заменяется на `100dvh` только там, где нужна высота экрана;
- нижние action-панели становятся sticky/flow или получают keyboard-aware режим.

Существующий `keyboardFocus` код сохраняется только если он помогает после foundation. Хаки, которые компенсируют старые fixed layout ошибки, удаляются после миграции соответствующих экранов.

## Скролл

Норма: один основной вертикальный scroll страницы.

Локальный scroll допустим только для:

- модального окна;
- dropdown;
- длинного списка внутри ограниченной панели;
- chat/message viewport, если он является основным смысловым scroll экрана.

`overflow: hidden`, `overflow-y: auto`, `height: 100%`, `max-height`, `calc(100vh - ...)` проверяются и удаляются там, где они создают вложенные scroll-зоны, пустоты или обрезание.

## Маршруты миграции

Переводятся все пользовательские и административные поверхности:

- профиль;
- статистика профиля;
- реферальная система;
- модули;
- уроки;
- общение;
- темы;
- чат;
- оплата;
- тарифы;
- поддержка;
- тикеты;
- уведомления;
- админка;
- клиенты;
- карточка клиента;
- рассылки;
- новая рассылка;
- платежи;
- хранилище;
- настройки проекта;
- админы;
- сервер;
- экраны пусто/загрузка/ошибка;
- модальные окна, dropdown, bottom sheet и fixed/sticky панели.

## Очистка CSS

CSS очищается по правилу: сначала новый foundation и миграция экрана, затем удаление конфликтов.

Удаляются или изолируются:

- повторные определения системных классов;
- локальные размеры action-кнопок меньше 44 px;
- локальные фиксированные высоты карточек и страниц;
- конфликтующие `min-height: 100vh`;
- случайные padding/margin вне токенов;
- глобальные one-off overrides, добавленные ради конкретных скриншотов;
- локальные цвета, нарушающие темы.

Не используется “глобальный `overflow-x: hidden` как лекарство”. Причины горизонтального overflow исправляются на уровне конкретных flex/grid/text правил.

## Тестирование

Добавляются и/или расширяются проверки:

1. Unit/Vitest:
   - foundation токены присутствуют;
   - базовые компоненты рендерят правильные роли/accessible names;
   - icon/action кнопки имеют системные размеры;
   - PageHeader переносит длинный текст.

2. Playwright:
   - все route-группы проходят геометрический аудит;
   - нет horizontal overflow;
   - кнопки не меньше 44 px;
   - значимые элементы не выходят за viewport;
   - bottom nav/action bar не перекрывают последний элемент;
   - keyboard focus сценарии для ticket/chat/forms.

3. Visual screenshots:
   - `320x720`;
   - `390x844`;
   - `768x1024`;
   - `1024x768`;
   - `1440x900`.

4. PWA sanity:
   - manifest;
   - service worker registration;
   - viewport-fit;
   - installable shell не ломается.

## Версия, кеш и деплой

После завершения foundation-миграции:

- обновляется версия приложения;
- обновляется timestamp версии;
- обновляется service worker cache name;
- сборка деплоится на сервер;
- production проверяется отдельно: версия, активный JS/CSS, service worker cache и ключевые маршруты.

## Готовность

Работа считается готовой только после свежих проверок:

- `pnpm -r test`;
- `pnpm -r check`;
- `pnpm -r build`;
- Playwright route audit;
- visual screenshot audit по обязательным viewport;
- ручная или автоматизированная проверка keyboard-сценариев;
- проверка production после деплоя.

Финальный отчёт должен перечислить изменённые foundation-компоненты, мигрированные маршруты, удалённые конфликтующие CSS-зоны, результаты build/lint/typecheck/tests, список viewport и известные ограничения.

## Уточнение плотности интерфейса после исправления Samsung desktop viewport

После устранения ошибочного desktop viewport `980px` прежние компенсирующие размеры стали визуально слишком крупными на физическом экране телефона. Пользователь утвердил компактный вариант примерно `0.8` от текущей визуальной плотности без глобального `transform`, `zoom` или уменьшения touch target.

Обновлённый контракт:

- page padding остаётся `16px`, а на ширине до `360px` — `12px`;
- section gap уменьшается с `24px` до `16px`;
- card gap уменьшается с `16px` до `12px`;
- card padding уменьшается с `20px` до `16px`;
- card radius уменьшается с `20px` до `18px`;
- обычные/крупные/компактные кнопки остаются `48/52/44px`;
- icon button остаётся `44x44px`, glyph уменьшается до `22px`;
- header, bottom navigation и bottom action получают высоты `68/68/64px`;
- действия PageHeader переходят на отдельную строку только до `380px`, а не до `480px`;
- чат сохраняет `16px` для поля ввода, но получает более компактные padding/radius/gap;
- scaled Samsung shell использует те же физически видимые значения после компенсации viewport.

Таким образом, визуальные поверхности становятся компактнее примерно на 20–25%, но интерактивные зоны сохраняют минимум `44x44px`.
