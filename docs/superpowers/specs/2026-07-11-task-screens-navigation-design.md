# Task Screens Navigation Design

## Цель

Заменить большие и интерактивные модальные окна полноценными экранами приложения. Сохранить текущие API, данные, права, авторизацию и бизнес-логику. Убрать постоянную ручную адаптацию высоты модалок под разные телефоны и экранную клавиатуру.

## Основной принцип

Каждый оверлей получает один из четырёх типов:

1. `Task screen` — отдельный маршрутизируемый экран для форм, длинных данных, файлов, редакторов и чатов.
2. `Bottom sheet` — короткий выбор из нескольких действий без длинного ввода.
3. `Dialog` — только короткое подтверждение одного действия.
4. `Media viewer` — полноэкранный просмотр изображения, видео или вложения.

Если сценарий содержит текстовое поле, загрузку файлов, несколько секций, длинный список или собственную прокрутку, он не должен быть модальным окном.

## Навигационная архитектура

Использовать `vue-router` как стандартный механизм маршрутов и истории браузера. Текущие разделы `profile`, `learning`, `community`, `payments`, `support`, `admin` становятся верхнеуровневыми маршрутами. Существующая логика выбора раздела сохраняется на переходном этапе через синхронизацию с маршрутом.

Маршруты task screen получают `meta.task = true`. На мобильном task screen скрывает нижнее меню и показывает собственную шапку с кнопкой назад. На desktop остаётся боковая навигация, а task screen занимает рабочую область приложения.

Общий компонент `TaskScreen` содержит:

- sticky header: назад, заголовок, необязательное главное действие;
- scrollable body в обычном потоке страницы;
- необязательный sticky footer для основных кнопок;
- safe-area отступы;
- единые состояния загрузки, ошибки и несохранённых изменений;
- ширину контента до 920 px на desktop и 100% на телефоне.

Кнопка Android Back, browser Back и системный жест iOS возвращают на предыдущий экран через router history. Прямая ссылка открывает тот же экран после восстановления сессии и проверки прав.

### Bottom sheets

- выбор платёжной системы перед подключением;
- выбор источника вложения или действия с файлом;
- короткие контекстные действия с сущностью, если они не содержат форму;
- инструкции системного действия, которое нельзя выполнить автоматически.

Bottom sheet занимает высоту по содержимому, имеет один scroll container и закрывается свайпом вниз, кнопкой назад или явной кнопкой закрытия. Поля длинного текста в sheet не размещаются.

### Компактные dialogs

- подтверждение выхода;
- подтверждение оплаты;
- подтверждение закрытия обращения;
- запрос разрешения на push;
- подтверждение удаления или другого необратимого действия.

Dialog содержит заголовок, короткое описание и не более двух основных действий. Высота всегда определяется содержимым. Внутренней прокрутки и сложных форм нет.

### Media viewers

- просмотр вложения поддержки;
- просмотр изображения урока;
- просмотр видео урока.

Viewer остаётся полноэкранным слоем, потому что его задача — изолированный просмотр медиа. Он не использует общие modal-размеры и закрывается стандартной кнопкой назад/закрыть.

## Поведение клавиатуры

Task screen использует обычный viewport приложения, поэтому не вычисляет высоту формы через modal CSS variables. При появлении клавиатуры:

- sticky header остаётся доступным;
- активное поле прокручивается в видимую область;
- footer располагается после содержимого или над клавиатурой без двойного вычитания inset;
- body имеет один вертикальный scroll container;
- нет backdrop, центрирования и transform у формы.

Специальная keyboard-логика остаётся только для чата, где поле сообщения действительно fixed.

## Сохранение состояния и данных

- API-вызовы, DTO и Pinia stores не меняются.
- Состояние формы переносится в компонент task screen или существующий feature store.
- Идентификаторы сущностей берутся из route params.
- После сохранения используется `router.back()` или переход к detail route.
- При ошибке пользователь остаётся на экране, а сообщение показывается рядом с действием.
- Для несохранённой формы Back вызывает единый compact dialog подтверждения выхода без сохранения.

## Права доступа

Router guards проверяют:

- наличие авторизованной сессии;
- активный доступ участника для закрытых разделов;
- роль администратора или владельца для admin routes;
- допустимость операции над конкретной сущностью после загрузки данных.

Недоступный deep link возвращает пользователя в разрешённый раздел с понятным сообщением, не показывая данные защищённого экрана.

## Визуальная система

Task screen использует существующий Soft Touch дизайн:

- фон приложения без затемняющего backdrop;
- секции как обычные surface-блоки;
- радиусы и токены текущей светлой/тёмной темы;
- header и footer с матовым blur только при прокрутке;
- touch targets не меньше 44 px;
- без горизонтального скролла на ширине 360–430 px.

## Порядок миграции

1. Добавить router, `TaskScreen`, guards и синхронизацию верхних разделов.
2. Перенести поддержку и рассылки как самые проблемные keyboard-сценарии.
3. Перенести клиентов, статистические drilldown, хранилище, сервер и права.
4. Перенести платёжные формы и тарифы.
5. Перенести редакторы модулей/уроков, просмотр урока, аватар и уведомления.
6. Оставить компактные dialogs, bottom sheets и media viewers.
7. Удалить неиспользуемые modal-классы, viewport-компенсации и дублирующий CSS.

Каждый этап сохраняет рабочий production: старый оверлей удаляется только после теста нового маршрута.

## Тестирование

- unit-тесты маршрутов, guards и возврата назад;
- deep-link тесты после восстановления сессии;
- проверка прав участника, администратора и владельца;
- Playwright на iPhone SE, iPhone Pro Max, Pixel, Galaxy, Huawei, OnePlus и wide-layout Android;
- клавиатура в поддержке, рассылках, платёжных формах и редакторах;
- отсутствие горизонтального скролла;
- сохранение/отмена формы и защита несохранённых изменений;
- светлая и тёмная тема;
- desktop sidebar и mobile back navigation;
- проверка, что в проекте остаются только разрешённые compact dialogs, sheets и media viewers.

## Критерии готовности

- ни одна длинная форма не открывается поверх текущего экрана;
- поддержка и рассылки не меняют размер при открытии клавиатуры;
- все task screens открываются по URL и корректно работают с Back;
- нижнее меню скрывается только внутри мобильного task screen;
- API, авторизация, данные и права работают без изменений контракта;
- старые универсальные правила размеров модалок удалены после полной миграции;
- полный unit/build/Playwright набор проходит перед production-деплоем.
