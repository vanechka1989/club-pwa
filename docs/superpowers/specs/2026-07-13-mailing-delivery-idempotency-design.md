# Защита рассылок от повторной доставки

## Проблема

Создание рассылки немедленно вызывает `processMailingQueue()`, а фоновый таймер вызывает ту же функцию каждые пять секунд. Пока медленная email-доставка ещё выполняется, следующий вызов повторно выбирает те же строки со статусом `pending`. В результате один пользователь получает несколько одинаковых push, уведомлений в приложении и писем, а `sentCount` становится больше `deliveryCount`.

Production-инцидент 13.07.2026 подтвердил причину: для 8 пользователей и 16 уникальных доставок база сохранила `sentCount = 41`, хотя таблица получателей содержала ровно 8 `push` и 8 `email` строк.

## Решение

Перед сетевой отправкой обработчик должен атомарно перевести конкретную строку получателя из `pending` в `processing` условным `UPDATE ... WHERE status = 'pending' RETURNING`. Отправку продолжает только обработчик, получивший строку. Параллельные обработчики пропускают уже захваченную доставку.

Счётчики рассылки увеличиваются атомарными SQL-выражениями, а статус `completed` выставляется только при отсутствии строк и в `pending`, и в `processing`.

Если отправка завершилась ошибкой, строка переходит из `processing` в `failed`; при недоступном адресате — в существующий статус `skipped_*`; при успехе — в `sent`.

## Не входит в изменение

- Повторная отправка уже продублированной рассылки.
- Изменение аудитории, фильтров или интерфейса создания рассылки.
- Локальный mutex, который не защищает несколько экземпляров API.

## Проверка

- Регрессионный тест требует атомарный захват `pending -> processing`.
- Тест требует учитывать `processing` при определении завершения.
- Тест требует атомарные приращения счётчиков.
- Полные тесты, TypeScript check и production build должны пройти.
- После выкладки production должен отдавать новую версию, health-check и серверный commit.
