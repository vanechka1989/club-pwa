# Push на устройстве и журнал IP-входов

Дата: 11.07.2026  
Статус: согласованная архитектура, ожидает финального подтверждения перед планом реализации

## Цели

1. Показывать фактическое состояние Web Push на текущем устройстве и позволять включать или отключать push именно для него.
2. Хранить уникальные IP-адреса клиента и даты их использования для контроля подозрительных входов.
3. Показывать журнал IP в карточке клиента только владельцу проекта и администраторам с отдельным правом, назначенным разработчиком.
4. Оформить экран уведомлений как выбранный вариант A: компактный список с понятной иерархией и иконкой корзины для очистки.

## Границы задачи

- Push отключается только на текущем браузере или установленном PWA.
- Системное разрешение браузера `Notification.permission` приложение отменить не может; оно отключает локальную `PushSubscription` и отзывает её на сервере.
- IP не используется как идентификатор устройства и не блокирует вход автоматически.
- Геолокация по IP, автоматические блокировки, уведомления о подозрительном входе и пользовательский экран «Мои устройства» в эту задачу не входят.
- API авторизации, существующие роли и бизнес-логика приложения сохраняются; добавляются только необходимые контракты безопасности.

## 1. Состояние и переключение push

### Источник истины

При открытии экрана уведомлений клиент получает регистрацию Service Worker и вызывает `registration.pushManager.getSubscription()`:

- подписка существует — состояние `enabled`;
- подписки нет — состояние `disabled`;
- API недоступно — `unsupported`;
- разрешение запрещено браузером — `denied`;
- во время проверки или операции — `checking`, `enabling` или `disabling`;
- ошибка синхронизации — `error` с понятным сообщением.

Состояние не определяется только по локальной переменной Pinia: оно заново синхронизируется с браузером после запуска экрана.

### Включение

1. Проверить поддержку Notification, Service Worker и PushManager.
2. Запросить разрешение, если оно ещё не выдано.
3. Получить существующую подписку или создать новую с VAPID-ключом.
4. Сохранить подписку через `POST /push/subscriptions`.
5. Показать кнопку «Отключить push» и уведомление «Push-уведомления включены на этом устройстве».

### Отключение текущего устройства

1. Получить текущую `PushSubscription`.
2. Передать её endpoint и ключи в `DELETE /push/subscriptions`, чтобы сервер выставил `revokedAt`.
3. Вызвать `subscription.unsubscribe()` в браузере.
4. Повторно проверить `getSubscription()` и показать состояние `disabled`.
5. Показать уведомление: «Push отключены. Оповещения больше не будут приходить на это устройство».

Отключение одной подписки не изменяет подписки пользователя на других устройствах.

### Обработка ошибок

- Кнопка блокируется только на время текущей операции, но остаётся читаемой.
- Если браузер уже не имеет подписки, состояние считается `disabled`; неизвестный локально endpoint не отзывается массово, а устаревшая серверная запись будет помечена отозванной после ответа push-службы `404/410`.
- При запрете системного разрешения показывается подсказка, что разрешение нужно вернуть в настройках браузера/PWA.
- Ошибка API не маскируется: состояние повторно считывается из браузера, пользователю показывается точный результат и возможность повторить действие.

## 2. Визуальный вариант A для уведомлений

- Сохраняется утверждённая компактная двухстрочная шапка.
- Push-действие содержит `BellPlus` и текст «Включить push» в состоянии `disabled`; в состоянии `enabled` — `BellOff` и текст «Отключить push».
- Очистка истории — отдельная компактная icon-кнопка с `Trash2`, доступным названием и зоной нажатия 44×44 px.
- Карточка уведомления компактная: статусная иконка 36 px, заголовок и дата в одной строке при достаточной ширине, текст ниже.
- Непрочитанное уведомление отмечается акцентной левой границей и не полагается только на цвет: используется соответствующая статусная иконка.
- Длинные заголовки и тексты переносятся, дата не сжимается, карточки не имеют фиксированной высоты.
- На ширине до 359 px push-действие и очистка переносятся ниже без наложений.
- Используются существующие семантические токены четырёх тем; локальные жёсткие цвета не добавляются.

## 3. Модель журнала IP

Добавляется таблица `user_login_ips`:

- `id` — UUID;
- `userId` — FK на пользователя с каскадным удалением;
- `ipAddress` — нормализованный точный IPv4 или IPv6;
- `firstSeenAt` — первое появление IP;
- `lastSeenAt` — последнее возвращение или переключение на IP;
- `loginCount` — число входов/переходов на этот IP;
- уникальный индекс `(userId, ipAddress)`;
- индекс `(userId, lastSeenAt)` для карточки клиента.

История хранится без автоматического срока удаления и удаляется каскадно вместе с пользователем.

В `auth_sessions` добавляется `lastIpAddress`. Он нужен, чтобы не обновлять журнал на каждом API-запросе.

### Правило записи

- После успешной проверки email-кода и создания сессии текущий IP записывается через upsert, а `auth_sessions.lastIpAddress` устанавливается этим IP.
- На каждом успешно аутентифицированном запросе IP сравнивается с `session.lastIpAddress`.
- Если IP совпадает, журнал не меняется.
- Если IP изменился, выполняется upsert `(userId, ipAddress)`: для нового адреса создаётся строка; для ранее встречавшегося обновляются `lastSeenAt` и `loginCount`. Затем обновляется `session.lastIpAddress`.
- Возврат с IP B на ранее использованный IP A обновляет существующую строку A, не создавая дубликат.
- Технические запросы без успешной авторизации в журнал клиента не попадают.

## 4. Безопасное получение IP

- API принимает трафик через Caddy внутри закрытой Docker-сети.
- Создаётся единый helper нормализации IP; дублирующая логика заголовков не размазывается по маршрутам.
- Caddy перезаписывает клиентские forwarding-заголовки перед передачей запроса API. Берётся адрес из этой доверенной proxy-границы; произвольная цепочка от клиента не считается доверенной.
- Удаляются пробелы, IPv4-mapped IPv6 нормализуется, порт не сохраняется, невалидные значения отклоняются.
- Если доверенный IP определить нельзя, авторизация продолжает работать, но журнал не записывается и ошибка фиксируется технически без раскрытия пользователю.

## 5. Право доступа

В `AdminPermission` добавляется отдельное право `security` с подписью «Безопасность клиентов».

- Владелец/разработчик получает его автоматически через `allAdminPermissions`.
- Новый администратор по умолчанию его не получает.
- Разработчик выдаёт или отзывает право в существующем экране управления администраторами.
- API возвращает IP-журнал только владельцу или активному администратору с `security`.
- Наличие общего права `users` само по себе не раскрывает IP.
- Клиентская карточка не рендерит блок и не запрашивает данные без `security`, но основной контроль остаётся на API.
- Просмотр IP-журнала не разрешается режимом предпросмотра роли клиента.

## 6. Карточка клиента

В карточке клиента появляется секция «Безопасность входов» только при наличии права:

- сверху последний использованный IP и время;
- далее остальные IP по `lastSeenAt DESC`;
- для каждой строки: точный IP, «Впервые», «Последний вход», количество входов;
- IP, впервые появившийся менее 24 часов назад, получает отметку «Новый IP» без отдельной серверной сущности тревоги;
- IPv6 переносится через `overflow-wrap: anywhere`;
- пустое состояние: «История IP появится после следующего входа клиента».

IP-журнал загружается защищённым endpoint карточки клиента или отдельным endpoint только после проверки `security`. Он не включается в общие списки клиентов и статистику.

## 7. Контракты и миграция

- Drizzle-схема и SQL-миграция создают `user_login_ips` и поле `auth_sessions.last_ip_address`.
- Shared-контракт добавляет `AdminLoginIp`, защищённый ответ истории и permission `security`.
- API push получает клиентскую функцию удаления подписки; существующий серверный `DELETE /push/subscriptions` используется и усиливается проверкой принадлежности endpoint текущему пользователю.
- Старые сессии с `lastIpAddress = null` безопасно зафиксируют IP при первом следующем аутентифицированном запросе.
- Старые push-записи и уведомления остаются совместимыми.

## 8. Проверки

### Автоматические

- Нормализация IPv4, IPv6 и IPv4-mapped IPv6.
- Новый IP создаёт запись; повторный IP не создаёт дубликат; возврат к старому IP обновляет его дату и счётчик.
- Неизменившийся IP не пишет журнал повторно.
- API скрывает журнал без `security` и отдаёт его владельцу/разрешённому админу.
- Push-state определяется из реальной подписки браузера.
- Включение сохраняет подписку; отключение отзывает серверную запись и вызывает `unsubscribe()`.
- Endpoint нельзя отозвать от имени другого пользователя.
- Кнопки меняют подпись и disabled/loading-состояние корректно.

### Визуальные

- Экран уведомлений: 320×640, 390×844 и 768×1024 в четырёх темах.
- Карточка клиента с одним, несколькими, длинными IPv6 и пустым журналом.
- Нет горизонтального overflow; иконка корзины и push-кнопка имеют зоны нажатия не меньше 44×44 px.

## Критерии готовности

- Фактическое состояние push корректно после перезапуска PWA.
- Подпись кнопки строго соответствует текущему устройству.
- Отключение не затрагивает другие устройства и сопровождается понятным уведомлением.
- Уникальная история IP обновляется по согласованному правилу без дубликатов.
- IP недоступны без отдельного права `security`.
- Выбранный компактный вариант A одинаково работает во всех четырёх темах и на мобильных viewport.
