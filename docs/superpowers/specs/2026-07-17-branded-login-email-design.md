# Фирменное письмо с кодом входа

## Цель

Заменить простое текстовое письмо с кодом входа на аккуратное HTML-письмо Club, которое одинаково читается в Gmail, Apple Mail, Outlook и мобильных почтовых клиентах. Письмо должно помогать быстро найти и перенести шестизначный код, не меняя существующую модель авторизации.

## Границы

- Изменяется только формирование письма с кодом входа.
- SMTP, срок действия кода, ограничение попыток, API авторизации и база данных не меняются.
- Код не добавляется в URL и не превращается в magic link.
- Текстовая версия письма сохраняется для клиентов, которые не показывают HTML.

## Визуальная структура

Письмо строится на табличной разметке и inline-стилях для совместимости с почтовыми клиентами:

1. Скрытый preheader: «Код для входа в Club. Действует 10 минут».
2. Светлый нейтральный фон письма.
3. Центральная карточка шириной до 560 px.
4. Шапка с иконкой Club 56×56 px и названием приложения.
5. Заголовок «Код для входа» и короткая инструкция.
6. Крупный моноширинный код с визуальным разделением `073 567`.
7. Подсказка: «Нажмите и удерживайте код, чтобы скопировать».
8. Кнопка «Открыть приложение», ведущая на `WEB_ORIGIN` без email и кода в URL.
9. Отдельный блок безопасности: срок действия и рекомендация проигнорировать письмо, если вход не запрашивали.
10. Небольшая подпись Club внизу.

Иконка загружается с `${WEB_ORIGIN}/icons/icon-192.png`. Если почтовый клиент блокирует изображения, `alt="Club"` и текстовый заголовок сохраняют смысл шапки.

## Поведение кода

JavaScript в письмо не добавляется: Gmail, Apple Mail и Outlook блокируют выполнение скриптов и доступ к Clipboard API. Вместо ненадёжной кнопки копирования код выводится отдельным крупным текстовым блоком с `user-select: all` как необязательным улучшением. На телефоне его можно выделить долгим нажатием, на компьютере — двойным нажатием или обычным выделением.

Код перед вставкой в HTML обязательно экранируется и дополнительно проверяется как шесть цифр. Для отображения он делится пробелом на две группы по три цифры, а текстовая версия сохраняет исходные шесть цифр без пробела.

## Архитектура

В `apps/api/src/auth/emailAuth.ts` остаётся одна публичная функция `buildEmailLoginMessage`. Она возвращает:

- `subject` — прежняя тема письма;
- `text` — существующий понятный fallback;
- `html` — новый совместимый HTML-шаблон.

Функция получает `code`, `expiresInMinutes` и `webOrigin`. Маршрут авторизации передаёт `env.WEB_ORIGIN`. Общий `sendEmail` уже поддерживает поле `html`, поэтому транспорт и SMTP-конфигурация не изменяются.

## Безопасность и ошибки

- Код, email и другие секреты не помещаются в ссылку.
- Внешних шрифтов, JavaScript, форм и трекеров нет.
- Ссылка формируется только из уже валидированного `WEB_ORIGIN`.
- При отключённых изображениях письмо остаётся полностью понятным.
- Если HTML не поддерживается, почтовый клиент показывает `text`.

## Тестирование

Автоматические тесты должны подтвердить:

- наличие HTML и текстового fallback;
- наличие исходного кода в тексте и визуально сгруппированного кода в HTML;
- экранирование динамических значений;
- наличие логотипа и безопасной ссылки на `WEB_ORIGIN`;
- отсутствие кода и email в URL;
- отсутствие `<script>`, Clipboard API и inline-обработчиков событий;
- сохранение текущей темы письма и срока действия;
- прохождение полного API test/typecheck/build набора.

После публикации выполняется production-проверка версии, health endpoint и формирование тестового письма без вывода реального кода в серверные логи сверх уже существующего dev-поведения.
