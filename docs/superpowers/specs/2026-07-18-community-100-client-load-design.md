# Нагрузочная проверка Club PWA: 100 одновременных клиентов

## Цель

Проверить на текущем production-сервере не абстрактные HTTP-запросы, а полный realtime-контур сообщества: 100 одновременно открытых SSE-подключений, доставку событий, задержки, переподключения и работу обычного API под параллельной нагрузкой. Тест не является обещанием ёмкости для любого пользовательского поведения: отчёт отдельно фиксирует проверенный профиль и его ограничения.

## Безопасность production

- Нагрузка создаётся с внешней машины, чтобы генератор не расходовал CPU VPS.
- Используется временная скрытая тема, доступная только администраторам.
- 100 клиентов аутентифицируются одной временной сессией владельца. Это точно воспроизводит 100 HTTP/SSE-соединений и нагрузку авторизации каждого запроса, но не 100 разных пользовательских ролей; это ограничение явно указывается в отчёте.
- Подключения наращиваются ступенями: 10, 25, 50, 75, 100.
- Все тестовые сообщения, тема и сессия удаляются после прогона, в том числе после ошибки.
- Максимальный профиль требует `CONFIRM_PRODUCTION_LOAD=YES`.

## Сценарии

1. **Подключение:** каждый клиент должен получить SSE-событие `ready`; сверяется `/api/metrics.realtimeSubscribers`.
2. **Обычная доставка:** сообщения отправляются с паузой; для каждого события измеряется задержка от окончания POST до получения клиентом.
3. **Залп:** серия сообщений отправляется параллельно; проверяются потери, дубликаты и p50/p95/p99/max.
4. **Переподключение:** часть клиентов принудительно отключается и подключается снова; все должны снова получить `ready` и последующее событие.
5. **Маршрутизация:** событие должно содержать идентификатор временной темы; чужие `topicId` учитываются отдельно и не считаются доставкой тестового события.
6. **HTTP-профиль:** 100 виртуальных клиентов обращаются к `/api/me`, уведомлениям, поддержке и списку тем с реалистичными паузами.
7. **Ресурсы:** до, во время пика и после снимаются health/readiness, runtime metrics, CPU/RAM контейнеров и состояние PostgreSQL.

## Метрики и критерии

- подключилось и получило `ready`: 100/100;
- доставка целевых SSE-событий: 100%, без дублей;
- SSE p95 до 500 мс, p99 до 1500 мс;
- HTTP errors менее 0,5%, p95 до 500 мс, p99 до 1500 мс;
- отсутствие перезапусков контейнеров, OOM, 5xx и ошибок БД;
- все переподключённые клиенты восстановились.

## Автоматическая остановка

Прогон прекращается, если health/readiness перестаёт отвечать, доля ошибок превышает 1%, p99 устойчиво превышает 3 секунды, RSS API превышает установленный предел, либо доставка контрольного события неполная. После остановки обязательно выполняется очистка.

## Результат

Генератор сохраняет машинный JSON и читаемый Markdown-отчёт: параметры, фактическое число клиентов, доставка и латентность по сценариям, HTTP-показатели, ресурсы, ошибки, результат очистки и честный вывод о текущем запасе.
