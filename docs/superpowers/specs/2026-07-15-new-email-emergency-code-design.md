# Аварийный код для нового email

## Цель

Владелец клуба может создать одноразовый код входа для корректного email, даже если пользователь с этим адресом ещё не завершил регистрацию и отсутствует в таблице `users`.

## Поведение

- Endpoint остаётся доступен только владельцу через существующий `rejectIfNotOwner`.
- Код записывается в стандартную таблицу `auth_email_login_codes` и проверяется обычным `/auth/email/verify`.
- При успешной проверке обычный поток `findOrCreateEmailUser` создаёт нового пользователя либо возвращает существующего.
- Все ранее неиспользованные коды данного email аннулируются перед созданием нового.
- Повторная генерация для одного email ограничивается 30 секундами независимо от наличия пользователя.
- В журнале сохраняются email и срок действия, но не сам код и не его хэш.
- Если пользователь уже существует, `targetUserId` и `targetTelegramId` заполняются; для нового email они остаются `null`.

## Ошибки

- Некорректный email возвращает 400.
- Повторная генерация раньше 30 секунд возвращает 429 с оставшимся временем.
- Отсутствие пользователя больше не является ошибкой.

## Проверка

API-тест должен подтверждать отсутствие обязательного `404`, поиск rate limit по email в metadata и nullable-цель аудита. Полный набор тестов и production-сборка должны оставаться зелёными.
