# Архитектура

Проект устроен как небольшая monorepo-структура, которую можно развивать без превращения первого экрана в один огромный файл.

## Пакеты

- `apps/api` - Bun + Hono API. Здесь живут конфигурация окружения, Telegram-аутентификация, доступ к базе данных и HTTP routes.
- `apps/web` - Vue 3 mini app. Здесь живут интеграция с Telegram WebApp, навигация и пользовательские экраны.
- `packages/shared` - Zod-схемы и TypeScript-типы, общие для API и frontend.
- `tests/e2e` - Playwright и axe smoke-тесты для оболочки mini app.

## Backend

API routes сгруппированы по продуктовым зонам:

- `me` - текущий Telegram-пользователь и статус подписки.
- `learning` - категории и обучающие материалы.
- `payments` - тарифы и будущие данные по оплатам.
- `subscriptions` - checkout и операции с подпиской.
- `support` - темы поддержки и будущие обращения.

API должен оставаться stateless, чтобы его можно было масштабировать горизонтально. PostgreSQL является основным источником данных. Redis/BullMQ можно добавить позже для общего rate limiting, фоновых задач и scheduled-задач по подпискам.

## База Данных

Основные таблицы:

- `users` - Telegram identity и поля профиля.
- `subscriptions` - статус подписки, данные платёжного провайдера и дата окончания доступа.
- `content_categories` - разделы обучения.
- `content_items` - обучающие материалы форматов text, photo и video.
- `support_tickets` - будущие обращения в поддержку внутри приложения.

Индексы уже заложены вокруг поиска по Telegram ID, поиска подписок, списков контента и очереди поддержки. Для 1000 зарегистрированных пользователей PostgreSQL более чем достаточно, если списки контента будут paginated, а медиа не будет отдаваться через API-процесс.

## Frontend

Feature-папки повторяют продуктовую структуру:

- `features/profile` - профиль клиента и статус подписки.
- `features/learning` - информация, категории обучения и previews материалов.
- `features/billing` - оплата и действия с подпиской.
- `features/support` - темы поддержки и будущая форма обращения.
- `features/app` - навигация и общая оболочка приложения.

Так каждый раздел можно будет спокойно заменить на более богатый экран без переписывания всего приложения.

## Цель По Нагрузке

Текущая цель - минимум 1000 зарегистрированных пользователей. Перед запуском нужно определить точный профиль нагрузки:

- daily active users;
- peak concurrent users;
- expected requests per second;
- средний размер контента и медиа;
- объём payment webhook событий.

Для первой production-версии фото и видео должны храниться в object storage/CDN. API должен отдавать metadata и signed/public URLs, а не стримить большие файлы сам.
